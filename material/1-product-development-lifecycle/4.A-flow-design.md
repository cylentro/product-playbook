# Flow Design Deep-Dive


::::present
### Flow Design Deep-Dive
Bridging the gap between User interaction and System execution.

**The Focus:**
* Mapping the human journey (UX Flow).
* Documenting the business logic (Brain Flow).
* Architecting the data handshake (System Flow).

> [!TIP]
> **Logic before Pixels.** Never design a screen until you understand the business rules governing it.
::::

In Module 4, we move from high-level concepts to concrete blueprints. To ensure no gaps exist between the user experience and the system's execution, the squad must document three distinct types of flows for every feature.

::::present
### The 3 Layers of Flow Design
Bridging the gap between the User, the Business, and the Machine.

* **User Flow (UX)**: The visual journey and interaction points.
* **Business Flow (Logic)**: The rules and decision gates ("The Brain").
* **System Flow (Tech)**: The API handshake and data movement.

> [!TIP]
> Do not start UI design until the Business Flow is signed off. If you don't know the logic, you can't design the screens.
::::

## The User Flow (The UX Perspective)

**Goal**: Map the visual journey and interaction points of the user.

::::present
### User Flow: The UX Perspective
Visualizing the journey across screens and decision points.

* **?[Happy Path](The ideal, error-free path a user takes to complete a task successfully.)**: The "Golden Path" to completion.
* **?[Circular Paths](Handling loops where users go back, cancel, or edit previous fields without breaking the session.)**: Handling "Back" and "Undo" actions.
* **Edge & Empty States**: No results, timeouts, or disconnects.

| Case Example | Anteraja Pro Flow |
| :--- | :--- |
| **Start** | User opens Dashboard. |
| **Action** | Taps on the Pro Progress Bar ($x/60$). |
| **Decision** | **ELIGIBLE?** â†’ Claim Reward<br/>**NOT ELIGIBLE?** â†’ Ship More. |
::::

The User Flow is a step-by-step visualization of the screens and actions a user takes to complete a task. Unlike a simple "Happy Path," a professional User Flow must account for the "messy" reality of app usage.

* **Happy Path**: The ideal sequence of events (e.g., User opens AMA â†’ Clicks Check Rates â†’ Inputs destination â†’ Sees price).
* **Circular Paths**: What happens when a user hits "Back," "Cancel," or tries to edit a previous field?
* **Error & Empty States**: What does the user see if there are no search results or if their internet disconnects mid-transaction?
* **Case Example (Anteraja Pro)**: * Start: User opens Dashboard.
  * **Action**: Taps on the Pro Progress Bar.
  * **Screen**: Detailed view of current order count vs. the 60-order goal.
  * **Decision**: User clicks "Claim Reward" (if eligible) or "Ship More" (if not).

## The Business Flow (The Logic Perspective)

Goal: Document the rules, decision gates, and "If/Then" logic that govern the product.

::::present
### Business Flow: The Logic Perspective
Documenting the "Brain" of the featureâ€”rules over pixels.

* **Decision Gates**: Diamond nodes (e.g., "Is user B2B or Retail?").
* **Status Changes**: State triggers (e.g., "Set Status = PRO if Orders >= 60").
* **Calculations**: Specific formulas (e.g., "Premium = 0.2% of Value").

> [!NOTE]
> **Example: Shipping Insurance Rules**
> * If Value < Rp500k â†’ Optional.
> * If Value > Rp1M â†’ Show "High Value" toast.
> * If Vendor A is Offline â†’ Route to Vendor B (**Aggregator Logic**).
::::

The Business Flow strips away the UI and focuses entirely on the logic. This is the "brain" of the feature. It ensures that the business requirements we defined in Ideation (Module 3) are actually enforced.

* **Decision Gates**: Diamond-shaped nodes in a flowchart where a choice is made (e.g., "Is user a Retail or B2B segment?").
* **Status Changes**: Triggers that move a user from one state to another (e.g., "Update user status to PRO if order count >= 60 in Q1").
* **Calculations**: Specific formulas used (e.g., "Insurance Premium = 0.2% of Declared Value").
* **Case Example (Shipping Insurance)**:
  * **Rule 1**: If item value < Rp500,000, insurance is optional.
  * **Rule 2**: If item value > Rp1,000,000, show a "High Value" warning toast.
  * **Rule 3**: If insurance vendor A is offline, route the request to vendor B (Aggregator Logic).

### Application/System Flow (The Tech Perspective)

Goal: Map the technical "handshake" between the frontend, backend, and third-party services.

::::present
### System Flow: The Tech Perspective
The bridge to Engineeringâ€”mapping the data handshake.

* **?[API Handshakes](The request/response cycle between systems, e.g., App calls POST /v1/insurance.)**: Endpoint triggers and payloads.
* **Database Triggers**: Background updates and state persistence.
* **Third-Party Integrations**: External partner communication (e.g., Biza CSV).

| Handshake Step | Transactional Logic |
| :--- | :--- |
| **Request** | App sends `ItemValue` + `Category` to API. |
| **Logic** | System checks "Self-Insurance" vs. "Vendor" eligibility. |
| **External** | Call Insurance Vendor API with payload. |
| **Response** | Save Policy ID and return **SUCCESS** to UI. |
::::

This is the bridge to Engineering. It documents how data moves between systems. Without this, a PM might design a feature that is impossible to build given our current API infrastructure.

* **API Handshakes**: Which endpoint is called when a button is clicked? (e.g., POST /v1/insurance/calculate).
* **Database Triggers**: What data is being saved or updated in the background?
* **Third-Party Integrations**: The flow of data to external partners (e.g., Biza CSV upload â†’ Validation Service â†’ Address Master Data â†’ Response).
* **Case Example (Shipping Insurance Aggregator)**:
  * **Request**: AMA App sends ItemValue and Category to our Insurance Service.
  * **Logic**: Insurance Service checks our "Self-Insurance" rules vs. "Vendor" rules.
  * **External Call**: Service calls the Insurance Vendor API with a payload.
  * **Response**: Vendor returns a policy ID; our system saves it and displays "Success" to the user.
